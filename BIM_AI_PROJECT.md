# BIM.AI — Руководящий документ проекта

## Что мы строим

BIM.AI — нейросетевая система, которая по текстовому описанию на русском или английском генерирует готовые BIM-семейства (.rfa) для Autodesk Revit. Фокус — инженерные сети (MEP): вентиляция, водоснабжение, электрика, пожаротушение.

### Как это работает для пользователя

Инженер пишет: "Приточный потолочный диффузор Ø200мм, расход 150 л/с, Systemair"
→ Система выдаёт скачиваемый .rfa файл с правильными коннекторами, параметрами, типоразмерами.

### Почему это нужно

Создание MEP-семейств в Revit — ручная работа на 2-8 часов каждое. На рынке нет инструментов для автоматической генерации MEP-семейств с правильными коннекторами и расчётными параметрами. BIMobject и аналоги — это библиотеки готовых семейств от производителей, но не генерация новых.

---

## Архитектура: 5 звеньев цепочки

Система работает как конвейер. Каждое звено зависит от предыдущего. Нельзя перескакивать.

```
Звено 1        Звено 2        Звено 3        Звено 4         Звено 5
ДАННЫЕ    →    ОБУЧЕНИЕ   →   ИНФЕРЕНС   →   КОНВЕРТАЦИЯ  →  ИНТЕРФЕЙС
(датасет)      (fine-tune)    (API)          (JSON→.rfa)     (веб/бот)
```

### Звено 1 — ДАННЫЕ (датасет для обучения)

**Что это:** Набор пар "текстовый запрос → JSON-спецификация семейства". Модель учится на этих примерах.

**Что нужно для минимума:** ~3000 пар (сейчас есть ~136). Для хорошего качества — 5000-10000.

**Откуда берутся данные:**
- Синтетический генератор (готов, модуль `mep_generator.py`) — основной источник объёма
- Парсинг реальных .rfa файлов (парсер готов) — для валидации реалистичности
- Скрапинг BIMobject (скрапер готов) — метаданные реального оборудования

**Готовые модули:**
- `mep_schema.py` — JSON-схема MEP-семейства (17 полей, 23 категории Revit, коннекторы с flow direction / system type / shape, расчётные параметры с формулами, классификация OmniClass/IFC)
- `mep_generator.py` — генератор синтетических семейств (15 шаблонов оборудования по 4 доменам, стандартные инженерные ряды размеров)
- `mep_prompts.py` — шаблоны промптов (10 групп, 3 уровня детализации, RU/EN, экспорт в Alpaca и ChatML)
- `mep_validator.py` — валидатор (12 правил: совместимость domain↔system, стандартные размеры, баланс потоков, физические пределы)

**Чего не хватает:**
- Масштабирование: больше вариаций параметров, больше промпт-перефразировок → цель 3000+ пар
- Аугментация: опечатки, смешанный RU/EN, инженерный жаргон, сокращения
- Валидация по реальным .rfa: прогнать парсер по 50-100 реальным файлам, сравнить со схемой, найти пропущенные поля
- Недостающие шаблоны: чиллеры, градирни, кабельные лотки, трансформаторы, ИБП

### Звено 2 — ОБУЧЕНИЕ (fine-tuning модели)

**Что это:** Дообучение CodeLlama 7B на нашем датасете через QLoRA, чтобы модель научилась генерировать валидный JSON по текстовому описанию.

**Техстек:** Unsloth + QLoRA, Vast.ai RTX 4090

**Готово:**
- Скрипт обучения QLoRA (из предыдущей работы, для дверей/окон)

**Нужно:**
- Адаптировать скрипт под MEP-формат данных
- Подобрать гиперпараметры под ~3000 пар (lr ~2e-4, 3-5 epochs, lora_r=16-32)
- Настроить валидацию на каждой эпохе (прогон через mep_validator)
- Метрики: % валидного JSON, % правильных коннекторов, % стандартных размеров

### Звено 3 — ИНФЕРЕНС (API для генерации)

**Что это:** Обученная модель работает как сервис — принимает текст, возвращает JSON.

**Техстек:** FastAPI + vLLM

**Нужно:**
- FastAPI-сервер с эндпоинтами: POST /generate (текст → JSON), POST /validate (JSON → отчёт)
- vLLM для быстрого инференса (batched, ~2-5 сек на запрос)
- Постпроцессинг: парсинг JSON из ответа модели, авто-коррекция типичных ошибок, валидация через mep_validator
- Retry-логика: если JSON невалидный → повторная генерация с уточнённым промптом

### Звено 4 — КОНВЕРТАЦИЯ (JSON → .rfa)

**Что это:** Превращение JSON-спецификации в реальный файл Revit-семейства.

**Техстек:** Dynamo (Python) или Revit API (C#)

**Нужно:**
- Dynamo-скрипт или C#-плагин, который:
  - Открывает правильный .rft шаблон (по категории и host_type)
  - Создаёт геометрию (экструзии, вращения по описанию из JSON)
  - Добавляет коннекторы с правильными domain, flow direction, system type, размерами
  - Создаёт параметры (shared/family, instance/type)
  - Вставляет формулы
  - Генерирует типоразмеры
  - Сохраняет как .rfa
- Альтернативный путь: headless Revit через Forge/APS API (облачный)

**Это самое сложное звено** — Revit API имеет много ограничений, особенно для MEP-коннекторов.

### Звено 5 — ИНТЕРФЕЙС (пользовательский)

**Что это:** Веб-приложение или Telegram-бот, где инженер вводит запрос и получает файл.

**Нужно (минимум):**
- Веб-форма: текстовое поле + кнопка "Сгенерировать"
- Отображение JSON-превью с подсветкой
- Кнопка "Скачать .rfa"
- История запросов

## Ключевые сложности MEP (почему это труднее архитектурных семейств)

1. **Коннекторы** — у каждого MEP-элемента есть точки подключения. Каждая имеет: domain (Duct/Pipe/Conduit/CableTray), flow direction (In/Out/Bidirectional), system type (SupplyAir, DomesticColdWater, Power...), shape и размеры. Без правильных коннекторов семейство бесполезно — оно не подключится к системе.

2. **Расчётные параметры** — расход воздуха (L/s), давление (Pa), мощность (W), напряжение (V), ток (A). Эти параметры участвуют в инженерных расчётах Revit. Неправильные единицы или диапазоны ломают расчёт.

3. **Формулы** — параметры зависят друг от друга. Диаметр подключения зависит от расхода. Мощность двигателя — от расхода и напора. Потери давления — квадратично от расхода. Модель должна генерировать корректные формулы.

4. **Стандартные размеры** — трубы идут в ряду DN15-DN600, воздуховоды — Ø100-Ø1250 с шагом по ГОСТ/DIN/SMACNA. Нестандартный размер = невозможно подключить к реальной системе.

5. **Совместимость систем** — нельзя подключить Pipe-коннектор к Duct, нельзя назначить SupplyAir системе DomainPiping. Валидатор это проверяет.

## Техстек

| Компонент | Технология | Зачем |
|---|---|---|
| Язык | Python 3.11+ | Основной язык проекта |
| LLM | CodeLlama 7B | Хорошо генерирует структурированный JSON, помещается на 1× RTX 4090 |

| Конвертация | Dynamo / Revit API | Создание .rfa из JSON |
| Парсинг .rfa | olefile + PartAtom XML | Чтение существующих семейств |
| Скрапинг | BIMobject API | Метаданные реального оборудования |

## Правила работы над проектом

1. **Модульность** — каждое звено разрабатывается и тестируется отдельно. Не начинаем следующее звено, пока предыдущее не работает.

2. **Валидация на каждом шаге** — любой сгенерированный JSON проходит через `mep_validator`. Невалидные данные не попадают в датасет.

3. **Реальные инженерные стандарты** — размеры, системы, классификация берутся из ГОСТ, DIN, ASHRAE, SMACNA, IFC. Не выдумываем.

4. **Двуязычность** — все промпты и интерфейс работают на русском и английском. Инженеры часто смешивают языки (ГВС = DHW, DN = Ду).

5. **Итеративность** — сначала минимально работающий прототип (один домен HVAC), потом расширение на остальные домены.

## Глоссарий

| Термин | Значение |
|---|---|
| .rfa | Файл семейства Revit (Revit Family) |
| .rft | Шаблон семейства Revit (Revit Family Template) |
| MEP | Mechanical, Electrical, Plumbing — инженерные сети |
| HVAC | Heating, Ventilation, Air Conditioning — ОВиК |
| FCU | Fan Coil Unit — фанкойл |
| AHU | Air Handling Unit — центральный кондиционер / приточная установка |
| VAV | Variable Air Volume — регулятор переменного расхода воздуха |
| DN | Diameter Nominal — условный диаметр трубы (DN50 ≈ 2") |
| CHW | Chilled Water — холодная вода (хладоноситель) |
| HW | Hot Water — горячая вода (теплоноситель) |
| CW | Cold Water — водопроводная холодная вода |
| QLoRA | Quantized Low-Rank Adaptation — метод дообучения LLM с 4-bit квантизацией |
| IFC | Industry Foundation Classes — открытый стандарт обмена BIM-данными |
| OmniClass | Система классификации строительных элементов |
